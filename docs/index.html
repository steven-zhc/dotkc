OpenClaw-first secrets workflow

Secrets you can run, not paste.

dotkc stores secrets in an encrypted vault file (ideal for iCloud Drive sync) and injects them into commands when you need them. It's built for personal OpenClaw/local-agent setups where you want secure, repeatable env injection without sprinkling secrets across .env files, shell history, or CI logs.

Get started npm source 
# initialize vault (creates iCloud Drive vault + local key file)
dotkc init

# store secrets in the vault (interactive hidden prompt)
dotkc set vercel acme-app-dev CLERK_SECRET_KEY

# inspect (redacted by default)
dotkc run vercel:acme-app-dev

# run your app with env injected
dotkc run vercel:acme-app-dev -- pnpm dev

Inspect mode prints redacted values by default. Use --unsafe-values only on a trusted machine.

Why dotkc?

The pain

  • .env files drift and leak (repos, backups, screenshots).
  • Copy/paste secrets into terminals ends up in history.
  • Agents (like OpenClaw) need secrets frequently, but you don't want them in chat.

The idea

  • Encrypt secrets into a vault file.
  • Sync the vault via iCloud Drive.
  • Inject them into commands at runtime.

Designed for OpenClaw

  • iCloud Drive sync is reliable and easy to reason about.
  • The vault is encrypted; iCloud only sees ciphertext.
  • A per-machine key file enables unattended OpenClaw runs.

How it works

dotkc stores secrets using a simple 3-dimension naming model: service + category + KEY.

Storage model

Secrets are stored in an encrypted vault file using: service + category + KEY

  • service: the SaaS or system (e.g. vercel, github)
  • category: project/environment (e.g. acme-app-dev)
  • KEY: env var name (e.g. CLERK_SECRET_KEY)

Tip: avoid : in category names to keep parsing unambiguous.

Two modes: run vs inspect

  • Run: dotkc run <spec> -- <cmd> injects env into the child process.
  • Inspect: dotkc run <spec> prints redacted values so you can sanity check quickly.

Want full values? Use --unsafe-values (and accept the logging risk).

OpenClaw pattern: manage secrets on your laptop → sync an encrypted vault via iCloud Drive → OpenClaw host uses dotkc run to inject secrets into tools (Next.js, REST services, CLIs, etc.).

Use cases

dotkc is optimized for personal workflows and agent automation. These are the patterns it was designed for.

OpenClaw / agent-friendly output

When an agent is calling dotkc, plain text output is annoying to parse and easy to misread. dotkc supports an OpenClaw-friendly JSON envelope via --openclaw (alias for --format openclaw). 

# let an agent discover the CLI surface area

dotkc --help --openclaw

# status as structured JSON

dotkc status --openclaw

This mode is for tooling: { format, command, ok, code, data, warnings, errors }.

Allowlist what an agent can request (spec file)

Best practice: commit a spec allowlist (no values) and let dotkc resolve values at runtime. This avoids sprinkling secret names across prompts and scripts. 

# dotkc.spec (checked into repo)
# one spec per line; comments allowed
fly.io:acme-app-dev
vercel:acme-app-dev:CLERK_SECRET_KEY

# inspect as OpenClaw JSON (redacted)
dotkc run --spec-file ./dotkc.spec --openclaw

# run your app with injected env
dotkc run --spec-file ./dotkc.spec -- pnpm dev

Local dev without leaking secrets

Prefer running commands with injected env rather than exporting env vars into your shell. 

dotkc run vercel:acme-app-dev -- pnpm dev

Find / reorganize secrets fast

# find a key (no values printed)
dotkc search clerk

# duplicate an environment (dev → prod)
dotkc copy fly.io:acme-app-dev fly.io:acme-app-prod

# rename/move categories
dotkc move fly.io:acme-app-prod vercel:acme-app-prod

Quickstart

Up and running in five steps.

Install

npm i -g dotkc
# or
pnpm add -g dotkc

Initialize vault

dotkc init

Creates the iCloud Drive vault (if missing) and a local key file (not synced). If they already exist, dotkc will ask whether to overwrite.

Import from .env

cd /path/to/your/project

dotkc import vercel acme-app-dev .env

Inspect secrets (redacted)

dotkc run vercel:acme-app-dev
# JSON for tools/agents
# dotkc run --format openclaw vercel:acme-app-dev

Run your app with injected env

dotkc run vercel:acme-app-dev -- pnpm dev
# Next.js build
# dotkc run vercel:acme-app-dev -- pnpm build

# exact (single key)
dotkc run vercel:acme-app-dev:CLERK_SECRET_KEY -- node ./app.mjs

Why a vault vs .env?

  • Encrypted at rest (AES‑256‑GCM). The synced file is ciphertext.
  • Less accidental leakage (repos, zip files, backups, screenshots, shell history).
  • Sync via iCloud Drive works well for personal laptop ↔ Mac mini setups.
  • Better agent hygiene: OpenClaw can use secrets without you pasting them into chat.

dotkc is optimized for personal workflows. For large teams and production secret management, consider dedicated platforms (Vault, AWS/GCP secrets, 1Password CLI, etc.).

FAQ

Is inspect mode safe?

It's safer than printing raw values: dotkc prints redacted values + length by default. Still, be mindful of terminal logs, screen recording, and shared machines.

How do I print the full values?

Use dotkc run --unsafe-values <spec> (inspect mode) or dotkc export <spec> --unsafe-values to export dotenv lines. dotkc will print a warning to stderr.

Does dotkc support Windows/Linux?

Vault backend is file-based (encrypted vault + local key file). It works well on macOS where iCloud Drive is available. On other OSes, behavior depends on whether you can sync the vault file and protect the local key file.

Does dotkc create backups?

Yes. Before overwriting the vault file, dotkc creates a backup (e.g. dotkc.vault.bak-20260213-165627415) and keeps the most recent 3 backups. You can configure this with DOTKC_BACKUP_KEEP (set 0 to disable) and DOTKC_BACKUP_DIR. If the backup fails, dotkc refuses to write the new vault.

What does dotkc store exactly?

Secrets are stored in an encrypted vault file. dotkc never stores secrets in your git repo unless you explicitly put them there. The per-machine local key file is required to decrypt the vault.

dotkc vs common secret tools

dotkc is intentionally small. Here’s how it compares in practice.

dotkc

  • Encrypted vault file + local key (great for 1–3 personal machines).
  • Optimized for env injection: dotkc run ... -- <cmd>.
  • Agent-friendly structured output (--openclaw).

1Password CLI (op)

  • Full password manager ecosystem: sharing, permissions, recovery, audit (best for teams).
  • Great integrations, but heavier setup and account dependency.

Cloud secrets (AWS/GCP)

  • Production-grade governance and IAM.
  • Overkill for personal local dev; requires cloud access and more config.

Notes

Machine B (SSH host) bootstrap

If your OpenClaw host is a separate machine, copy the key file over (securely) and install it: 

# on machine A
cat ~/.dotkc/key | ssh user@machine-b 'dotkc key install'

# verify on machine B
ssh user@machine-b 'dotkc status'

# if something is wrong
ssh user@machine-b 'dotkc doctor'

If you run dotkc init on machine B, it will create a NEW key that cannot decrypt an existing vault encrypted with a different key.

Backups + sync conflicts (P0 safety)

Before overwriting the vault, dotkc creates a backup and refuses to write if backup fails. If the vault file changes on disk during an operation (iCloud sync conflict), dotkc refuses to overwrite and asks you to retry. 

# defaults
DOTKC_BACKUP_KEEP=3

# disable backups
DOTKC_BACKUP_KEEP=0

# store backups in a separate directory
DOTKC_BACKUP_DIR=/path/to/dir

Backup filenames look like dotkc.vault.bak-YYYYMMDD-HHMMSSmmm.

Best practices

  • Keep category stable and descriptive (project + env).
  • Prefer dotkc run <service>:<category> -- <cmd> instead of exporting env in your shell.
  • Use --unsafe-values only on a trusted personal machine.
  • Back up and protect the local key file; anyone with vault + key can decrypt.
  • Use dotkc search to locate keys across services/categories without printing values.
  • Use dotkc copy / dotkc move to duplicate or rename an environment (e.g. dev → prod) without re-entering values.
  • For agents, use dotkc run --format openclaw <spec> to get a structured JSON payload.
  • Use --spec-file to keep a checked-in allowlist of required secrets (no values) for an agent workflow.
